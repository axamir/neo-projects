# حاکمیت پروژه

نسخه: ۰.۱

وضعیت: سند زنده

---

# هدف

حاکمیت پروژه مشخص می‌کند که تصمیم‌گیری، مالکیت، رضایت، انتشار و مسئولیت در پروژه‌های Lewis OS چگونه مدیریت می‌شوند.

هدف این است که ابهام پیش از تبدیل‌شدن به اختلاف برطرف شود.

---

# اصول اصلی حاکمیت

- مالکیت باید صریح باشد.
- رضایت باید مستند شود.
- سکوت به‌معنای رضایت نیست.
- محتوای خصوصی به‌طور پیش‌فرض خصوصی باقی می‌ماند.
- ارتباط عمومی افراد با پروژه نیازمند تأیید است.
- تصمیم‌های مهم باید ثبت شوند.
- تاریخچه پروژه باید قابل پیگیری باشد.
- هوش مصنوعی می‌تواند کمک کند، اما مسئولیت نهایی انسانی است.

---

# نقش‌ها

## بنیان‌گذار

فردی که ایده اصلی را معرفی، مالکیت یا هدایت می‌کند.

مسئولیت‌های احتمالی:

- تعریف نیت
- بازبینی تصمیم‌های اصلی
- تأیید ادعاهای عمومی
- حفاظت از جهت پروژه
- تأیید مرزهای مالکیت

## معمار

فردی که ایده را به یک سیستم پروژه ساختاریافته تبدیل می‌کند.

مسئولیت‌های احتمالی:

- معماری پروژه
- مستندسازی
- طراحی گردش کار
- شناسایی ریسک
- توسعه نقشه راه
- برنامه‌ریزی تداوم

## سازنده

فرد یا تیم مسئول پیاده‌سازی.

مسئولیت‌های احتمالی:

- اجرای فنی
- طراحی
- تست
- استقرار
- به‌روزرسانی مستندات
- گزارش موانع

## مشاور

فردی که دانش تخصصی ارائه می‌دهد، بدون آنکه خودکار مالکیت یا اختیار دریافت کند.

## سرمایه‌گذار

فرد یا نهادی که بر اساس قرارداد جداگانه سرمایه وارد می‌کند.

## سیستم‌های هوش مصنوعی

هوش مصنوعی می‌تواند در تحقیق، نگارش، کدنویسی، بازبینی، تحلیل و خودکارسازی کمک کند.

سیستم‌های AI مالک دارایی‌های پروژه نیستند و جایگزین تأیید انسانی نمی‌شوند.

---

# مالکیت

مالکیت باید به‌صورت جداگانه برای موارد زیر تعریف شود:

- ایده اولیه
- نام پروژه
- برند
- کد منبع
- مستندات
- دارایی‌های طراحی
- مدل کسب‌وکار
- داده‌ها
- دامنه‌ها
- ریپوها
- خروجی‌های تولیدشده
- آثار مشترک

هیچ انتقال مالکیتی بدون توافق کتبی فرض نمی‌شود.

---

# رضایت

پیش از موارد زیر، تأیید صریح لازم است:

- انتشار نام شخص
- معرفی فرد به‌عنوان بنیان‌گذار، شریک یا مشاور
- استفاده از گفت‌وگوهای خصوصی
- انتشار تاریخچه شخصی
- اشتراک‌گذاری اطلاعات محرمانه
- انتقال ریپو
- صدور مجوز دارایی‌های پروژه
- فروش یا واگذاری حقوق پروژه

رضایت می‌تواند با توجه به تعهدات حقوقی موجود اصلاح یا پس گرفته شود.

---

# طبقه‌بندی حریم خصوصی

محتوای پروژه باید در یکی از این سطح‌ها قرار گیرد:

## خصوصی

فقط برای افراد تأییدشده قابل مشاهده است.

## داخلی

برای تیم کاری قابل دسترسی است.

## قابل بازبینی

برای بررسی آماده شده اما عمومی نیست.

## عمومی

برای انتشار تأیید شده است.

## دارای مجوز

استفاده از آن تحت شرایط مشخص مجاز است.

---

# ثبت تصمیم‌ها

تصمیم‌های مهم باید شامل این موارد باشند:

- تصمیم
- تاریخ
- افراد حاضر
- شواهد بررسی‌شده
- گزینه‌های جایگزین
- وضعیت تأیید
- پیامدها
- زمان بازبینی بعدی

تصمیم‌ها نباید فقط داخل تاریخچه چت باقی بمانند.

---

# نسخه‌بندی

هر سند مهم باید شامل این موارد باشد:

- شماره نسخه
- وضعیت
- تاریخ آخرین به‌روزرسانی
- منبع یا مالک
- وضعیت تأیید

تغییرات اصلی باید در تاریخچه Git قابل پیگیری باشند.

---

# انتشار

پروژه فقط زمانی منتشر می‌شود که:

- مالکیت روشن باشد
- محتوای حساس حذف شده باشد
- ادعاهای عمومی بازبینی شده باشند
- نام‌ها و نقش‌ها تأیید شده باشند
- لینک‌ها و منابع بررسی شده باشند
- وضعیت فعلی پروژه دقیق نمایش داده شود

---

# انتقال ریپو

زمانی که پروژه به ریپوی مستقل منتقل می‌شود، بسته تحویل باید شامل این موارد باشد:

- اسناد منبع تأییدشده
- تاریخچه ریپو یا منشأ مستند
- بیانیه مالکیت
- اعتباردهی به مشارکت‌کنندگان
- مجوز
- بریف پیاده‌سازی
- نقشه راه
- Backlog
- مجوزهای دسترسی
- مرزهای حریم خصوصی

نسخه اولیه پروژه می‌تواند به‌عنوان سابقه تاریخی در Neo Projects باقی بماند.

---

# حل اختلاف

اختلاف‌ها باید از این مسیر مدیریت شوند:

1. شفاف‌سازی کتبی
2. بررسی توافق‌های موجود
3. گفت‌وگوی مبتنی بر شواهد
4. ثبت راه‌حل پیشنهادی
5. میانجی‌گری در صورت نیاز
6. بررسی حقوقی در موارد ضروری

# قانون حاکمیت

هیچ پروژه‌ای نباید سریع‌تر از ظرفیت ساختار مالکیت، رضایت و مسئولیت خود رشد کند.
